# ChatGPT2LocalBridge Roadmap

Objective: make Web ChatGPT feel like it can safely operate an approved local workspace by calling clear MCP tools instead of guessing shell commands.

## Roadmap Rules

- `[x]` means the item has current evidence in code, docs, tests, or a recorded release.
- `[ ]` means planned, partial, unverified, blocked, or still needs user input.
- Low-level tools stay available for advanced debugging, but product docs and tool descriptions should guide ChatGPT toward safer mid-level and high-level tools first.
- Public deployments should default to OAuth or Secure MCP Tunnel. No Authentication is only for loopback, private lab, or short-lived test tunnels.
- Do not mark broad safety goals complete until the tool catalog, runtime policy, app UI, and deployment docs all match.

## Current Baseline

- The bridge already exposes local file, project, policy, shell, and activity tools through MCP.
- Web ChatGPT can call the connector and record tool activity.
- The current rough edge is tool selection: ChatGPT can overuse `shell.exec`, `file.write`, and many small low-level calls when a single higher-level task tool would be safer.
- The product needs a cleaner split between debug primitives, workflow tools, and Codex-powered agent tools.

## Key Decisions

- Keep low-level tools, but label them as advanced/debug tools.
- Promote mid-level project tools for reading context, validating policy, inspecting diffs, and running tests.
- Route complex work through structured handoffs before local Codex execution. See [Handoff Spec](./docs/handoff-spec.md).
- Add high-level `codex.*` tools so Web ChatGPT can delegate local software work to Codex CLI through a controlled runner.
- Treat local skills as a safe registry first, then a manifest-driven runtime. See [Skill Runtime Roadmap](./docs/skill-runtime-roadmap.md).
- Default public guidance to OAuth or Secure MCP Tunnel.
- Disable or hide raw `shell.exec` in normal user mode when a safer tool can handle the same workflow.
- Make the native/local app the operator console for policy, traces, running tasks, logs, diffs, and cancellation.

## Tool Tiers

### Low-Level Debug Tools

Purpose: direct primitives for advanced troubleshooting.

- `file.read_path`
- `file.write`
- `shell.exec`

Policy:

- [ ] Mark these tools as advanced/debug in descriptions, docs, and the app catalog.
- [ ] Add warnings when ChatGPT calls `shell.exec` for tasks that should use `codex.task_start`, `test.run`, or `project.bundle`.
- [ ] Support a normal mode where `shell.exec` is disabled or omitted from the public connector tool list.
- [ ] Keep an explicit debug mode for trusted local development.

### Mid-Level Project Tools

Purpose: inspect and operate a project without forcing ChatGPT to compose shell commands.

- `project.bundle`
- `policy.read`
- `git.diff`
- `test.run`

Policy:

- [ ] Make `project.bundle` the recommended first call for multi-file context.
- [ ] Add or harden `git.diff` so ChatGPT can inspect local changes without raw shell.
- [ ] Add or harden `test.run` so common test commands are policy-controlled and traceable.
- [ ] Ensure every mid-level tool writes a structured tool-call record.
- [ ] Add examples that tell ChatGPT to use mid-level tools before low-level tools.

### High-Level Codex Runner Tools

Purpose: give Web ChatGPT one clean entry point for local coding work.

Initial tool set:

- `handoff.create`
- `codex.task_start`
- `codex.status`
- `codex.result`

Target tool set:

- [x] `handoff.create`: validate and persist a structured task handoff without executing code.
  - Evidence: 2026-06-21 `docs/handoff-spec.md`, `src/mcpServer.ts`, and `scripts/test-mcp.sh`.
- [ ] Harden `codex.task_start`: start a bounded local Codex CLI job in an approved root with production-grade runner controls.
- [ ] Harden `codex.status`: return lifecycle state, elapsed time, active step, and last log lines.
- [ ] Harden `codex.result`: return summary, changed files, diff, tests, and artifacts.
- [ ] `codex.cancel`: stop a running job from ChatGPT or the local app.
- [ ] `codex.tail`: stream or page recent logs for long jobs.
- [ ] `codex.review_diff`: ask local Codex CLI to review current changes without applying edits.

Runner constraints:

- [ ] Require `projectRoot` to be inside `allowedProjectRoots`.
- [ ] Enforce per-task timeout and output limits.
- [x] Run Codex CLI with the intended local runner permission mode.
  - Current mode: `--sandbox danger-full-access --dangerously-bypass-approvals-and-sandbox`.
  - Boundary still comes from bridge policy, handoff constraints, timeout, logs, and App trace.
- [ ] Store task logs, stdout/stderr, generated patches, and final summaries in the runtime data directory.
- [ ] Record all runner lifecycle events in `tool-calls.jsonl` and audit logs.
- [ ] Expose changed-file and diff summaries before any public or destructive operation.

## Security Plan

### Public Access

- [ ] Use OAuth for normal public tunnel deployments.
- [ ] Use OpenAI Secure MCP Tunnel when available to avoid exposing a public MCP URL.
- [ ] Keep No Authentication limited to loopback, private lab, or temporary test endpoints.
- [ ] Document that a no-auth public tunnel is a capability URL and should be treated as sensitive.

### Shell Safety

- [ ] Make `shell.exec` opt-in for normal deployments.
- [ ] Add a policy switch for `publicToolProfile`: `normal`, `debug`, or `codex-runner-only`.
- [ ] In normal mode, prefer `codex.*`, `project.bundle`, `git.diff`, and `test.run`.
- [ ] Apply deny patterns before shell execution and record blocked attempts.
- [ ] Add app UI controls for enabling/disabling shell access with visible risk copy.

### Policy Editing

- [ ] Move policy editing into the local app UI.
- [ ] Validate roots, deny globs, skill roots, shell settings, and auth mode before saving.
- [ ] Warn when broad roots such as `/`, `$HOME`, `~/.codex`, or `/Volumes` are approved.
- [ ] Show exactly which roots are visible to ChatGPT.
- [ ] Add one-click export of the active policy without secrets.
